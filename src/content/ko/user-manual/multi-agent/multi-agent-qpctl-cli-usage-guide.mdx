# Multi Agent - qpctl CLI 사용 가이드

### Overview

`qpctl`은 QueryPie Multi Agent와 함께 제공되는 CLI 도구입니다.
Seamless SSH 연결뿐 아니라, Multi Agent가 관리하는 QueryPie Host, SAC Role, 인증 세션을 명령줄에서 제어할 수 있습니다.

#### qpctl 명령 구성

| **분류** | **명령**            | **설명**                                                     |
| -------- | ------------------- | ------------------------------------------------------------ |
| SSH      | `qpctl ssh`         | OpenSSH 명령을 래핑하여 QueryPie 경유 SSH 연결을 수행합니다. |
| SSH      | `qpctl ssh-proxy`   | SSH 프록시 연결을 수행합니다.                                |
| Host     | `qpctl host list`   | 등록된 QueryPie Host 목록을 조회합니다.                      |
| Host     | `qpctl host use`    | 사용할 QueryPie Host를 선택합니다.                           |
| Role     | `qpctl role list`   | 사용 가능한 SAC Role 목록을 조회합니다.                      |
| Role     | `qpctl role use`    | 현재 사용할 SAC Role을 변경합니다.                           |
| Auth     | `qpctl auth login`  | QueryPie 인증 세션을 시작합니다.                             |
| Auth     | `qpctl auth logout` | QueryPie 인증 세션을 종료합니다.                             |
| Auth     | `qpctl auth status` | 현재 인증 상태를 확인합니다.                                 |

<Callout type="info">
Role, Auth 인자는 11.5.6 부터 지원됩니다.
</Callout>

### 시작하기 전에

다음 조건을 먼저 확인합니다.

* QueryPie Multi Agent가 설치되어 있어야 합니다.
* `qpctl` 명령을 실행할 수 있어야 합니다.
* QueryPie Multi Agent가 실행 중이어야 합니다.
* `host` 명령을 사용하려면 앱에 QueryPie Host가 하나 이상 등록되어 있어야 합니다.
* `role` 명령을 사용하려면 사용자에게 SAC Role이 하나 이상 할당되어 있어야 합니다.
* `auth login`을 사용하려면 QueryPie 사용자 계정이 필요합니다.

#### qpctl 설치 방식

소스 기준으로 `qpctl`은 QueryPie Multi Agent와 별도로 받는 독립 패키지라기보다, QueryPie Multi Agent와 함께 제공되는 CLI입니다.

* macOS: Multi Agent 앱 내부의 `qpctl` 실행 파일을 `/usr/local/bin/qpctl`에 symbolic link로 설치합니다.
* Linux: 배포 패키지에 `qpctl`이 포함되며 release 기준 설치 경로는 `/usr/bin/qpctl`입니다.
* Windows: 설치 프로그램이 `qpctl.exe`를 시스템 디렉터리에 함께 설치합니다.

따라서 일반 사용자 관점에서는 `qpctl`을 따로 설치하기보다 QueryPie Multi Agent를 설치한 뒤 `qpctl` 명령이 PATH에서 실행되는지 확인하면 됩니다.

예시:
```bash
qpctl --help
```

명령이 실행되지 않으면 QueryPie Multi Agent 설치 상태 또는 CLI 링크/경로 설정을 먼저 확인합니다.

### 기본 사용 흐름

`qpctl` 직접 사용 흐름은 크게 다음과 같습니다.

1. QueryPie Host를 정합니다.
2. `qpctl ssh-proxy` 또는 `qpctl ssh`를 사용해 QueryPie 경유 SSH 연결을 수행합니다.
3. 필요하면 SSH config 또는 shell alias에 반영하여 반복 사용합니다.
4. 필요 시 `qpctl auth login`으로 CLI 인증 상태를 확인하고 로그인합니다.
5. 필요 시 `qpctl role list`, `qpctl role use`로 사용할 SAC Role을 조회하고 전환합니다.

### QueryPie Host 관리하기

#### `qpctl host list`

등록된 QueryPie Host 목록을 출력합니다.
```bash
qpctl host list
```

출력 동작:

* 등록된 Host가 없으면 `No QueryPie hosts configured.`를 출력합니다.
* 현재 선택된 Host는 `(current)` 표시와 함께 출력합니다.

예시:
```text
QueryPie Hosts:
 - https://nightly.dev.querypie.io
 - http://t8.ec2.querypie.io (current)
 - https://t3.qa.querypie.io
```

<Callout type="important">
여기서 보이는 `(current)`는 `qpctl`이 기본으로 사용할 QueryPie Host를 의미합니다.
</Callout>

`qpctl host use` 명령으로 사용할 host를 지정할 수 있습니다.

##### Host를 직접 지정하는 경우

```bash
qpctl host use https://develop.dev.querypie.io
```

성공하면 다음 형식으로 출력합니다.
```text
✔ Host changed to https://develop.dev.querypie.io
```

##### Host를 지정하지 않는 경우

```bash
qpctl host use
```

이 경우 등록된 Host 목록을 interactive 선택 화면으로 보여줍니다.

예시:
```text
Use the arrow keys to navigate: ↓ ↑ → ←
QueryPie Hosts
  ▸ https://nightly.dev.querypie.io
    http://t8.ec2.querypie.io (current)
    https://t3.qa.querypie.io
```

### SSH 연결에 사용하는 qpctl 명령

#### `qpctl ssh-proxy`

`qpctl ssh-proxy`는 SSH `ProxyCommand`에 연결되는 프록시용 명령입니다.

사용 형식:
```bash
qpctl ssh-proxy <user> <host> <port> [QueryPie Host]
```

예시:
```bash
qpctl ssh-proxy ec2-user 10.0.0.10 22
qpctl ssh-proxy ec2-user 10.0.0.10 22 https://develop.dev.querypie.io
```

인자 설명:

| **인자**        | **설명**                        |
| --------------- | ------------------------------- |
| `user`          | SSH 사용자명                    |
| `host`          | SSH 대상 호스트명               |
| `port`          | SSH 대상 포트                   |
| `QueryPie Host` | 사용할 QueryPie Host. 선택 입력 |

동작 방식:

* `QueryPie Host`를 직접 지정하면 해당 값을 사용합니다.
* 지정하지 않으면 현재 선택된 기본 Host를 사용합니다.

#### SSH config에서 사용하는 예시

다음과 같이 SSH config에서 사용할 수 있습니다.
```text
Host {{Server Name}}
  Hostname {{Server URL}}
  Port {{Server SSH Port}}
  ProxyCommand qpctl ssh-proxy %r %h %p develop.dev.querypie.io
```

또는 기본 Host를 미리 선택해 둔 경우 다음처럼 간소화할 수 있습니다.
```bash
qpctl host use develop.dev.querypie.io
```

그 뒤 `ProxyCommand`는 다음처럼 사용할 수 있습니다.
```text
ProxyCommand qpctl ssh-proxy %r %h %p
```

#### `qpctl ssh`

`qpctl ssh`는 OpenSSH 명령줄을 받아 QueryPie 경유 연결을 수행하는 래퍼 명령입니다.

기본 예시:
```bash
qpctl ssh root@ssh.server.io
```

추가 OpenSSH 옵션도 함께 전달할 수 있습니다.
```bash
qpctl ssh -L 8080:localhost:80 root@ssh.server.io
qpctl ssh -i ~/.ssh/id_rsa root@ssh.server.io
```

동작 방식:

* 내부적으로 SSH 설정을 해석합니다.
* 필요한 `ProxyCommand`를 자동으로 붙여 실제 `ssh` 프로세스를 실행합니다.

제약 사항:

* 사용자가 직접 `ProxyCommand` 옵션을 지정하는 방식은 지원하지 않습니다.
* `-G` 플래그가 포함된 경우에는 자동 ProxyCommand 주입 없이 원래 SSH 동작을 그대로 수행합니다.

<Callout type="important">
`qpctl ssh-proxy`는 SSH 설정의 `ProxyCommand`에 연결하는 방식이고, `qpctl ssh`는 OpenSSH 호출 자체를 감싸는 직접 실행 방식입니다.
</Callout>

### Role 명령

11.5.6부터는 CLI에서 SAC Role을 조회하고 변경할 수 있습니다.

#### `qpctl role list`

사용 가능한 SAC Role 목록을 출력합니다.
```bash
qpctl role list
```

출력 동작:

* Role이 없으면 `No roles found.`를 출력합니다.
* Role이 있으면 `UUID`, `Name`, `Description` 순서로 출력합니다.
* Description이 없는 Role은 `UUID`, `Name`만 출력합니다.

예시:
```text
Roles:
 - 11111111-1111-1111-1111-111111111111	DBA	Database administrators
 - 22222222-2222-2222-2222-222222222222	ReadOnly
```

##### 이름으로 필터링하는 경우

```bash
qpctl role list --name DBA
```

`--name` 값은 이름 완전일치가 아니라 포함 검색 방식으로 적용됩니다.

#### `qpctl role use`

현재 사용할 SAC Role을 변경합니다.

##### Role UUID를 직접 지정하는 경우

```bash
qpctl role use 11111111-1111-1111-1111-111111111111
```

성공하면 다음 형식으로 출력합니다.
```text
✔ Role changed to 11111111-1111-1111-1111-111111111111
```

##### 이름으로 Role을 찾는 경우

```bash
qpctl role use --name DBA
```

이 경우 이름 필터링 후 interactive 선택을 통해 Role을 고르게 됩니다.

##### 인자를 생략하는 경우

```bash
qpctl role use
```

이 경우 사용 가능한 Role 전체 목록을 interactive 선택으로 보여줍니다.

<Callout type="important">
`role-uuid`와 `--name`은 동시에 사용할 수 없습니다.
둘 다 지정하면 오류가 발생합니다.
</Callout>

### Auth 명령

11.5.6부터는 CLI에서 QueryPie 인증 세션 상태를 직접 확인하고 로그인/로그아웃할 수 있습니다.

#### `qpctl auth status`

현재 로그인 상태를 확인합니다.
```bash
qpctl auth status
```

출력 동작:

* 로그인되지 않은 경우: `Not logged in`
* 로그인된 경우 사용자명이 있으면: `Logged in as {username}`
* 로그인된 경우 사용자명이 없으면: `Logged in`

#### `qpctl auth login`

QueryPie 계정으로 로그인합니다.
```bash
qpctl auth login --username brendon
```

동작 방식:

1. 현재 인증 상태를 확인합니다.
2. 이미 로그인되어 있으면 새 로그인 대신 오류를 반환합니다.
3. 로그인되지 않은 상태라면 터미널에 `Password:` 프롬프트를 표시합니다.
4. 비밀번호는 화면에 노출되지 않는 방식으로 입력받습니다.
5. 인증에 성공하면 `Logged in as {username}`을 출력합니다.

예시:
```text
Password:
Logged in as brendon
```

#### `qpctl auth logout`

현재 인증 세션을 종료합니다.
```bash
qpctl auth logout
```

성공하면 다음 형식으로 출력합니다.
```text
Logged out
```

<Callout type="important">
이미 로그인된 상태에서 `auth login`을 다시 실행하면, 먼저 `qpctl auth logout`을 수행하라는 오류가 반환될 수 있습니다.
</Callout>

### 추천 사용 흐름

#### 1. 기본 Host를 먼저 지정하는 경우

```bash
qpctl host list
qpctl host use
qpctl auth status
qpctl auth login --username brendon
qpctl role list
qpctl role use --name DBA
qpctl ssh admin@db-server
```

이 흐름은 다음 상황에 적합합니다.

* 여러 QueryPie Host 중 하나를 선택해 반복 사용할 때
* CLI에서 인증 및 Role 상태를 먼저 정리한 뒤 SSH 연결까지 이어서 수행할 때

#### 2. SSH config 기반 Seamless SSH를 유지하는 경우

```text
Host {{Server Name}}
  Hostname {{Server URL}}
  Port {{Server SSH Port}}
  ProxyCommand qpctl ssh-proxy %r %h %p
```

이 경우에도 먼저 아래 명령으로 기본 Host를 지정해 두면 관리가 편합니다.
```bash
qpctl host use develop.dev.querypie.io
```

그 후 필요하면 추가로 다음 명령을 사용합니다.
```bash
qpctl auth status
qpctl role list
qpctl role use --name ReadOnly
```

즉 SSH 연결과 함께, 11.5.6부터는 인증과 Role 제어도 CLI에서 함께 수행할 수 있습니다.

### 문제 해결

#### `No QueryPie hosts configured.`가 출력되는 경우

* 앱에 QueryPie Host가 등록되어 있지 않은 상태입니다.
* 먼저 Multi Agent 앱에서 Host를 추가해야 합니다.

#### `No roles found.`가 출력되는 경우

* 현재 사용자에게 사용 가능한 SAC Role이 없거나,
* `--name` 필터 결과가 없는 경우입니다.

#### `auth login`이 실패하는 경우

다음을 확인합니다.

* `--username` 값을 입력했는지
* 비밀번호를 비워 두지 않았는지
* 이미 로그인된 상태는 아닌지
* Multi Agent가 정상 실행 중인지

#### `ssh` 명령에서 `ProxyCommand option is not supported` 오류가 나는 경우

* `qpctl ssh`는 사용자가 직접 `ProxyCommand`를 지정하는 방식을 지원하지 않습니다.
* `ProxyCommand` 없이 일반 OpenSSH 인자만 넘겨 사용해야 합니다.

### 참고 사항

* `role` 명령은 코드상 SAC product를 대상으로 동작합니다.
* `auth login`은 별도 신규 백엔드 비밀번호 로그인 endpoint를 추가하는 방식이 아니라, 기존 인증 흐름을 재사용하는 구조입니다.
